# 内容研究撰写助手

此技能作为你的写作伙伴，帮助你研究、列大纲、起草和完善内容，同时保持你独特的声音和风格。

## 何时使用此技能

- 撰写博客文章、文章或新闻通讯
- 创建教育内容或教程
- 起草思想领导力文章
- 研究和撰写案例研究
- 制作带来源的技术文档
- 撰写带正确引用和参考的内容
- 改进开头和介绍
- 在写作时获得逐节反馈

## 此技能的功能

1. **协作大纲制定**：帮助你将想法结构化为连贯的大纲
2. **研究协助**：查找相关信息并添加引用
3. **开头改进**：增强你的开篇以吸引注意力
4. **章节反馈**：在你写作时审查每个部分
5. **声音保持**：维持你的写作风格和语调
6. **引用管理**：正确添加和格式化参考
7. **迭代完善**：通过多个草稿帮助你改进

## 如何使用

### 设置你的写作环境

为你的文章创建专用文件夹：
```
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title
```

创建你的草稿文件：
```
touch article-draft.md
```

从此目录打开 Claude Code 并开始写作。

### 基本工作流程

1. **从大纲开始**：
```
帮我为一篇关于 [主题] 的文章创建大纲
```

2. **研究并添加引用**：
```
研究 [特定主题] 并在我的大纲中添加引用
```

3. **改进开头**：
```
这是我的介绍。帮我让开头更有吸引力。
```

4. **获得章节反馈**：
```
我刚完成"为什么这很重要"部分。审查它并给出反馈。
```

5. **完善和润色**：
```
审查完整草稿的流畅性、清晰度和一致性。
```

## 说明

当用户请求写作协助时：

1. **了解写作项目**

   询问澄清问题：
   - 主题和主要论点是什么？
   - 目标受众是谁？
   - 期望的长度/格式是什么？
   - 你的目标是什么？（教育、说服、娱乐、解释）
   - 任何要包含的现有研究或来源？
   - 你的写作风格是什么？（正式、对话、技术）

2. **协作大纲制定**

   帮助构建内容：

   ```markdown
   # 文章大纲：[标题]

   ## 开头
   - [开篇语句/故事/统计数据]
   - [读者为什么要关心]

   ## 介绍
   - 背景和上下文
   - 问题陈述
   - 本文涵盖的内容

   ## 主要章节

   ### 第 1 节：[标题]
   - 要点 A
   - 要点 B
   - 示例/证据
   - [需要研究：特定主题]

   ### 第 2 节：[标题]
   - 要点 C
   - 要点 D
   - 需要数据/引用

   ### 第 3 节：[标题]
   - 要点 E
   - 反论点
   - 解决方案

   ## 结论
   - 主要观点总结
   - 行动号召
   - 最后想法

   ## 研究待办
   - [ ] 查找关于 [主题] 的数据
   - [ ] 获取 [概念] 的示例
   - [ ] 为 [主张] 寻找引用来源
   ```

   **迭代大纲**：
   - 根据反馈调整
   - 确保逻辑流畅
   - 识别研究空白
   - 标记需要深入探讨的章节

3. **进行研究**

   当用户请求研究某个主题时：

   - 搜索相关信息
   - 查找可信来源
   - 提取关键事实、引述和数据
   - 以请求的格式添加引用

   示例输出：
   ```markdown
   ## 研究：AI 对生产力的影响

   主要发现：

   1. **生产力提升**：研究显示内容创作任务节省 40% 时间 [1]

   2. **采用率**：67% 的知识工作者每周使用 AI 工具 [2]

   3. **专家引述**："AI 增强而非取代人类创造力" - Jane Smith 博士，MIT [3]

   引用：
   [1] McKinsey Global Institute. (2024). "生成式 AI 的经济潜力"
   [2] Stack Overflow 开发者调查 (2024)
   [3] Smith, J. (2024). MIT Technology Review 访谈

   已添加到大纲第 2 节下。
   ```

4. **改进开头**

   当用户分享介绍时，分析并加强：

   **当前开头分析**：
   - 哪些有效：[积极元素]
   - 哪些可以更强：[改进领域]
   - 情感影响：[当前 vs 潜力]

   **建议替代方案**：

   选项 1：[大胆声明]
   > [示例]
   *为何有效：[解释]*

   选项 2：[个人故事]
   > [示例]
   *为何有效：[解释]*

   选项 3：[令人惊讶的数据]
   > [示例]
   *为何有效：[解释]*

   **开头问题**：
   - 是否创造好奇心？
   - 是否承诺价值？
   - 是否足够具体？
   - 是否匹配受众？

5. **提供逐节反馈**

   当用户撰写每个部分时，审查：

   ```markdown
   # 反馈：[章节名称]

   ## 哪些做得好 ✓
   - [优点 1]
   - [优点 2]
   - [优点 3]

   ## 改进建议

   ### 清晰度
   - [具体问题] → [建议修复]
   - [复杂句子] → [更简单的替代]

   ### 流畅性
   - [过渡问题] → [更好的连接]
   - [段落顺序] → [建议重新排序]

   ### 证据
   - [需要支持的主张] → [添加引用或示例]
   - [笼统陈述] → [使其更具体]

   ### 风格
   - [语调不一致] → [更好地匹配你的声音]
   - [词汇选择] → [更强的替代]

   ## 具体行编辑

   原文：
   > [草稿中的确切引用]

   建议：
   > [改进版本]

   原因：[解释]

   ## 需要考虑的问题
   - [发人深省的问题 1]
   - [发人深省的问题 2]

   准备进入下一节！
   ```

6. **保持作者声音**

   重要原则：

   - **学习他们的风格**：阅读现有写作样本
   - **建议而非替换**：提供选项，而非指令
   - **匹配语调**：正式、随意、技术、友好
   - **尊重选择**：如果他们更喜欢自己的版本，支持它
   - **增强而非覆盖**：让他们的写作更好，而非不同

   定期询问：
   - "这听起来像你吗？"
   - "这是正确的语调吗？"
   - "我应该更/更少 [正式/随意/技术]？"

7. **引用管理**

   根据用户偏好处理参考：

   **行内引用**：
   ```markdown
   研究显示 40% 的生产力提升 (McKinsey, 2024)。
   ```

   **编号参考**：
   ```markdown
   研究显示 40% 的生产力提升 [1]。

   [1] McKinsey Global Institute. (2024)...
   ```

   **脚注样式**：
   ```markdown
   研究显示 40% 的生产力提升^1

   ^1: McKinsey Global Institute. (2024)...
   ```

   维护运行中的引用列表：
   ```markdown
   ## 参考

   1. 作者. (年份). "标题". 出版物.
   2. 作者. (年份). "标题". 出版物.
   ...
   ```

8. **最终审查和润色**

   当草稿完成时，提供全面反馈：

   ```markdown
   # 完整草稿审查

   ## 总体评估

   **优势**：
   - [主要优势 1]
   - [主要优势 2]
   - [主要优势 3]

   **影响**：[整体效果评估]

   ## 结构与流畅性
   - [组织评论]
   - [过渡质量]
   - [节奏评估]

   ## 内容质量
   - [论证强度]
   - [证据充分性]
   - [示例效果]

   ## 技术质量
   - 语法和机制：[评估]
   - 一致性：[评估]
   - 引用：[完整性检查]

   ## 可读性
   - 清晰度评分：[评估]
   - 句子多样性：[评估]
   - 段落长度：[评估]

   ## 最终润色建议

   1. **介绍**：[具体改进]
   2. **正文**：[具体改进]
   3. **结论**：[具体改进]
   4. **标题**：[如需要的选项]

   ## 发布前检查清单
   - [ ] 所有主张都有来源
   - [ ] 引用已格式化
   - [ ] 示例清晰
   - [ ] 过渡流畅
   - [ ] 存在行动号召
   - [ ] 已校对错别字

   准备发布！🚀
   ```

## 示例

### 示例 1：Teresa Torres 的工作流程

**用户**："我正在写一篇关于连续发现的文章。帮我创建一个大纲。"

**过程**：
1. 协作制定大纲结构
2. 识别研究需求
3. 用户开始写介绍
4. 审查并改进开头
5. 用户撰写每个部分
6. 在每个部分后提供反馈
7. 进行研究并添加引用
8. 完整草稿的最终审查
9. 润色并准备发布

**结果**：以 Teresa 的声音撰写的、研究充分、正确引用的文章，具有强大的结构和流畅性。

### 示例 2：重研究文章

**用户**："我正在写关于 AI 对产品管理的影响。帮我查找当前数据和示例。"

**输出**：
```markdown
## 汇编的研究

### 近期数据
- Gartner：到 2025 年 80% 的 PM 将使用 AI 工具 [1]
- 调查：45% 报告功能发布更快 [2]
- 案例研究：Airbnb 的 AI 辅助优先级排序 [3]

### 专家引述
- "AI 放大 PM 的判断，而非取代它" - Marty Cagan
- [带引用的额外引述]

### 真实示例
1. **公司 A**：使用 AI 进行用户研究综合
   - 结果：节省 60% 时间
   - 来源：[引用]

2. **公司 B**：AI 驱动的路线图分析
   - 结果：更好的利益相关者对齐
   - 来源：[引用]

所有内容已添加到你的大纲，带有正确的引用。
```

### 示例 3：开头改进

**用户的原始开头**：
> "产品管理因 AI 而改变。在本文中，我将讨论 AI 影响产品经理的一些方式。"

**改进选项**：

**选项 1（数据驱动）**：
> "上个月，我让 AI 分析了 500 个客户访谈。它花了 30 分钟而不是 3 周。产品管理将永远不同。"

**选项 2（问题）**：
> "如果你能在早餐前与每个客户交谈、阅读每条评论并分析每个支持工单会怎样？"

**选项 3（故事）**：
> "Sarah 花了两周时间构建错误的功能。不是因为她不了解用户，而是因为她无法足够快地处理数百个访谈以发现模式。"

### 示例 4：章节反馈

**用户**："刚完成我的'常见错误'部分。想法？"

**响应**：
```markdown
# 反馈：常见错误部分

## 哪些做得好 ✓
- 强大的示例（Slack 通知故事）
- 清晰的结构（3 个错误，干净布局）
- 每个错误的实用建议

## 建议

### 使其更具体
你的第二点说"糟糕的优先级排序"但可以更强：

当前：
> "许多团队优先级排序糟糕，专注于功能而非结果。"

建议：
> "我看到团队在一个季度发布 14 个功能，却没有推动任何关键指标。他们优先考虑活动而非进步。"

### 添加数据
第三个错误将受益于证据：
> "[添加引用]：研究显示没有定期用户联系的团队构建未使用功能的可能性高 3 倍 [需要来源]"

### 流畅性改进
考虑重新排序：错误 3 → 错误 2 → 错误 1
这从小影响构建到大影响。

准备下一节！
```

## 写作工作流程

### 博客文章工作流程
1. 一起制定大纲
2. 研究关键点
3. 写介绍 → 获得反馈
4. 写正文部分 → 每节反馈
5. 写结论 → 最终审查
6. 润色和编辑

### 新闻通讯工作流程
1. 讨论开头想法
2. 快速大纲（较短格式）
3. 一次会话中起草
4. 审查清晰度和链接
5. 快速润色

### 技术教程工作流程
1. 列出步骤大纲
2. 编写代码示例
3. 添加解释
4. 测试说明
5. 添加故障排除部分
6. 最终审查准确性

### 思想领导力工作流程
1. 构思独特角度
2. 研究现有观点
3. 发展你的论点
4. 以强 POV 写作
5. 添加支持证据
6. 构思引人入胜的结论

## 专业提示

1. **在 VS Code 中工作**：比网页版 Claude 更适合长篇写作
2. **一次一个部分**：增量获得反馈
3. **单独保存研究**：保留 research.md 文件
4. **版本化你的草稿**：article-v1.md、article-v2.md 等
5. **大声朗读**：使用反馈识别笨拙的句子
6. **设置截止日期**："我想今天完成草稿"
7. **休息一下**：写作、获得反馈、暂停、修改

## 文件组织

写作项目的推荐结构：

```
~/writing/article-name/
├── outline.md          # 你的大纲
├── research.md         # 所有研究和引用
├── draft-v1.md         # 第一稿
├── draft-v2.md         # 修订稿
├── final.md            # 可发布版本
├── feedback.md         # 收集的反馈
└── sources/            # 参考材料
    ├── study1.pdf
    └── article2.md
```

## 最佳实践

### 用于研究
- 引用前验证来源
- 尽可能使用近期数据
- 平衡不同观点
- 链接到原始来源

### 用于反馈
- 具体说明你想要什么："这是否太技术？"
- 分享你的担忧："我担心这部分拖沓"
- 提问："这流畅逻辑吗？"
- 请求替代方案："解释这个的另一种方式是什么？"

### 用于声音
- 分享你的写作示例
- 指定语调偏好
- 指出好的匹配："那听起来像我！"
- 标记不匹配："对我的风格来说太正式"
